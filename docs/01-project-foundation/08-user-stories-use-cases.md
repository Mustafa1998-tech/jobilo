## Use Cases

### UC-1: User Registration

| Element | Description |
|---------|------------|
| **ID** | UC-1 |
| **Name** | تسجيل مستخدم جديد |
| **Actors** | زائر (غير مسجل) |
| **Preconditions** | - |
| **Postconditions** | حساب جديد نشط بعد تأكيد البريد |
| **Trigger** | المستخدم يضغط "اشترك" |

**Main Flow:**
1. المستخدم يختار نوع الحساب (Freelancer/Client)
2. النظام يعرض نموذج التسجيل (الاسم، البريد، كلمة السر، تأكيد كلمة السر)
3. المستخدم يدخل البيانات ويوافق على الشروط
4. النظام يتحقق من صحة البيانات (validation)
5. النظام يتحقق من عدم وجود البريد مسبقاً
6. النظام ينشئ الحساب (حالة: غير مفعل)
7. النظام يرسل بريد تأكيد مع OTP
8. المستخدم يدخل OTP
9. النظام يفعل الحساب
10. النظام يعرض رسالة ترحيب ويدخل المستخدم للمنصة

**Alternative Flows:**
- **AF-1**: Google OAuth
  1. يختار "تسجيل بـ Google"
  2. تنبثق نافذة Google Auth
  3. المستخدم يوافق على الصلاحيات
  4. النظام يستلم بيانات Google
  5. إذا كان البريد موجوداً → تسجيل دخول
  6. إذا جديد → إنشاء حساب مع بيانات Google
  
- **AF-2**: البريد موجود بالفعل
  1. النظام يظهر رسالة "البريد مسجل مسبقاً، هل ترغب في تسجيل الدخول؟"
  2. المستخدم يتجه لتسجيل الدخول

- **AF-3**: OTP لا يصل
  1. المستخدم يضغط "إعادة إرسال"
  2. النظام يرسل OTP جديد (حد 3 محاولات)

**Business Rules:**
- البريد الإلكتروني يجب أن يكون فريداً
- كلمة السر ≥ 8 أحرف، حرف كبير، رقم
- الحد الأدنى للعمر 18 سنة
- 3 محاولات OTP كحد أقصى كل 15 دقيقة

**Validation Rules:**
| Field | Rule | Message |
|-------|------|---------|
| Email | Required, valid format | "البريد الإلكتروني مطلوب" |
| Password | Min 8 chars, 1 upper, 1 number | "يجب أن تتكون كلمة السر من 8 أحرف على الأقل..." |
| Name | Required, 2-50 chars | "الاسم مطلوب" |
| Confirm Password | Must match password | "كلمتا السر غير متطابقتين" |
| Terms | Must accept | "يجب الموافقة على الشروط" |

### UC-2: Create Project

| Element | Description |
|---------|------------|
| **ID** | UC-2 |
| **Name** | إنشاء مشروع جديد |
| **Actors** | عميل (Client - مسجل، موثق البريد) |
| **Preconditions** | المستخدم مسجل دخول + نوع حسابه Client |
| **Postconditions** | مشروع جديد بحالة Draft أو Published |
| **Trigger** | العميل يضغط "نشر مشروع" |

**Main Flow:**
1. العميل يضغط على "نشر مشروع" في الـ Navigation
2. النظام يعرض نموذج إنشاء المشروع
3. العميل يدخل: العنوان، الوصف، التصنيف، المهارات المطلوبة
4. العميل يحدد: نوع الميزانية، المبلغ، المدة، مستوى الخبرة
5. العميل يرفع ملفات المشروع (اختياري)
6. العميل يضغط "معاينة"
7. النظام يعرض معاينة المشروع
8. العميل يضغط "نشر" أو "حفظ كمسودة"
9. النظام ينشر المشروع بحالة Open
10. النظام يرسل إشعارات للمستقلين المناسبين (AI matching)

**Alternative Flows:**
- **AF-1**: حفظ كمسودة
  1. العميل يضغط "حفظ كمسودة"
  2. النظام يحفظ المشروع بحالة Draft
  3. العميل يمكنه العودة للتعديل لاحقاً

- **AF-2**: AI اقتراحات
  1. النظام يحلل الوصف ويقترح تحسينات
  2. النظام يقترح نطاقاً سعرياً مناسباً
  3. العميل يوافق/يعدل

**Business Rules:**
- يجب أن يكون عنوان المشروع فريداً
- حد أدنى للميزانية: $10
- حد أقصى للمرفقات: 5 ملفات، 20MB لكل ملف
- يمكن نشر 3 مشاريع مجانية/شهر (Free clients)

| Field | Validation |
|-------|-----------|
| Title | Required, 10-200 chars |
| Description | Required, 50-5000 chars |
| Category | Required, must exist |
| Budget | Required, numeric, min $10 |
| Duration | Required, 1-365 days |

### UC-3: Submit Proposal

| Element | Description |
|---------|------------|
| **ID** | UC-3 |
| **Name** | تقديم عرض على مشروع |
| **Actors** | مستقل (Freelancer) |
| **Preconditions** | المستقل مسجل + المشروع بحالة Open |
| **Postconditions** | عرض جديد بحالة Pending |
| **Trigger** | المستقل يضغط "تقديم عرض" |

**Main Flow:**
1. المستقل يفتح صفحة تفاصيل المشروع
2. يضغط "تقديم عرض"
3. النظام يعرض نموذج العرض
4. المستقل يكتب: cover letter، السعر، المدة
5. AI يقترح عرضاً أولياً (اختياري)
6. المستقل يرفع ملفات (اختياري)
7. المستقل يستعرض العرض
8. المستقل يضغط "إرسال العرض"
9. النظام يتحقق من صحة البيانات
10. النظام يحفظ العرض بحالة Pending
11. النظام يرسل إشعاراً للعميل

**Business Rules:**
- لا يمكن تقديم عرضين على نفس المشروع
- السعر المقترح يجب أن يكون ضمن نطاق ميزانية المشروع (تحذير فقط)
- الحد الأقصى: 100 عرض لكل مشروع

### UC-4: Payment & Escrow

| Element | Description |
|---------|------------|
| **ID** | UC-4 |
| **Name** | إدارة الدفع والضمان |
| **Actors** | عميل، مستقل، نظام دفع |
| **Preconditions** | عقد موقع بين الطرفين |
| **Postconditions** | مبلغ في escrow / محرر للمستقل |

**Main Flow (Deposit):**
1. بعد توقيع العقد، النظام يعرض للعميل خيارات الدفع
2. العميل يدخل معلومات الدفع
3. نظام الدفع يتحقق من صحة المعلومات
4. يتم خصم المبلغ من حساب العميل
5. النظام يضيف المبلغ إلى حساب Escrow
6. النظام يحدث حالة الدفع إلى "Funded"

**Main Flow (Release):**
1. المستقل يسلم العمل
2. العميل يؤكد الاستلام
3. النظام يحرر المبلغ من Escrow
4. النظام يضيف المبلغ إلى محفظة المستقل (بعد خصم العمولة)
5. النظام يرسل إشعاراً للمستقل

**Edge Cases:**
- رفض التسليم → Dispute
- انتهاء صلاحية رابط الدفع → إعادة المحاولة
- فشل الدفع → رسالة خطأ مع إعادة المحاولة

### UC-5: Dispute Resolution

| Element | Description |
|---------|------------|
| **ID** | UC-5 |
| **Name** | حل النزاعات |
| **Actors** | عميل، مستقل، مسؤول |
| **Preconditions** | مشروع بحالة In Progress أو Under Review |
| **Postconditions** | مشروع بحالة Completed / Cancelled / Refunded |

**Main Flow:**
1. أحد الطرفين يضغط "فتح نزاع"
2. الطرف يشرح سبب النزاع مع الأدلة
3. النظام يوجه الطرف الآخر للرد
4. الطرف الآخر يقدم رده
5. النظام يحول النزاع لمسؤول
6. المسؤول يراجع الأدلة
7. المسؤول يتواصل مع الطرفين
8. المسؤول يصدر قراراً
9. النظام ينفذ القرار (تحرير/استرداد المبلغ)

**Business Rules:**
- Fee للنزاع: $5 (يسترد للفائز)
- مدة الرد: 48 ساعة
- إذا لم يرد أحد الطرفين، يحكم لصالح الآخر
- يمكن التصعيد إلى محكم خارجي
